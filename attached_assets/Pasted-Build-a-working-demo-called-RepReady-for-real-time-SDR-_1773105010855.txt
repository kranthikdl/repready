Build a working demo called “RepReady” for real-time SDR coaching during live sales calls.

Product intent:
RepReady is a real-time AI coaching demo for SDR readiness. It should support a live or simulated sales call, detect coaching-relevant moments, surface short coaching prompts during the session, and generate a post-call summary plus a simple readiness scorecard. The demo must feel structured, fast, and believable, not enterprise-complete.

Primary goal:
Build a bounded v0 demo that proves this workflow end to end:
1. User starts a session
2. User selects call type and coaching priorities
3. User speaks into microphone OR runs a simulated transcript feed
4. Transcript updates live on screen
5. Coaching engine detects moments and surfaces short prompts in a side panel
6. At end of session, app generates post-call summary, flags, and simple scorecard
7. Session artifact is saved and viewable

Important constraints:
- This is a DEMO, not a production app
- Optimize for clarity, speed, and demo stability
- Keep architecture simple and modular
- Do not add CRM integrations, auth, multi-tenancy, analytics warehouse, or heavy admin features
- UI must be minimally distracting
- Avoid prompt spam
- All coaching rules must be configurable without large code rewrites
- Use a sidecar coaching model, not an embedded meeting product

Recommended stack:
- Frontend: React + Vite + TypeScript
- Backend: Node.js + Express + TypeScript
- Realtime transport: WebSocket
- Styling: Tailwind
- Persistence: simple local JSON or SQLite
- LLM: OpenAI API
- Speech input:
  - Preferred: browser microphone capture + chunk upload/transcription path if feasible
  - Fallback: simulated transcript streaming mode that is fully demoable
- Do not block the whole build on perfect live speech transcription. If browser audio streaming is messy, provide both modes:
  1. “Live Demo Mode” using browser microphone/transcription pipeline
  2. “Simulation Mode” using scripted transcript chunks

Coaching panel requirements:

C. End session output
When user ends session:
- Generate structured post-call summary with sections:

  - Discovery
  - Objection handling

  - Overall readiness

- Provide a previous sessions screen to review saved sessions

Architecture and logic design:
Use a rules-plus-model hybrid.

1. Transcript ingestion module
- Receives transcript chunks from either:
  - live audio transcription path
  - simulation feed
- Normalizes chunks into a consistent event shape

2. Coaching engine
Split into two layers:

Layer 1: deterministic rule detection
- Detect potential coaching moments from transcript patterns
- Example heuristic rules:
  - If rep talks for too long without asking a question during discovery, raise missed discovery candidate
  - If buyer expresses hesitation with phrases like “not sure,” “already using,” “budget,” “timing,” raise objection candidate
  - If call approaches ending without commitment language, raise weak next step candidate

Layer 2: LLM refinement
- Given recent transcript window + candidate event + session config, generate:
  - whether prompt should fire
  - prompt title
  - one-line coaching recommendation
  - severity
- Keep outputs short and structured JSON only

3. Session controller
- Handles state
- Prevents prompt spam
- Maintains cooldowns
- Tracks active session
- Stores transcript, prompts, flags, summary, and scorecard

4. Summary generator
- At end of session, analyze full transcript + fired coaching events
- Return structured JSON for summary and scorecard
